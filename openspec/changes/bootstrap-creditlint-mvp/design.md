## Context

`creditlint` starts from an empty repository. The first implementation should
therefore establish both the product contract and the technical foundation.

The user-facing problem is not "AI usage exists"; it is that tools can place
AI/tool identifiers in authorship-like metadata without explicit maintainer
approval. The design needs to preserve a distinction between authorship signals
such as `Co-authored-by` and provenance/disclosure signals such as
`AI-Assisted`.

The implementation should be a Rust native CLI managed with Cargo. `creditlint`
is intended to run in many repositories, including backend, systems, infra, and
polyglot projects that may not have Node.js, npm, or pnpm installed. A native
binary keeps the consuming-project integration surface small. pnpm remains
useful only as a local runner for OpenSpec commands during planning.

## Goals / Non-Goals

**Goals:**

- Provide a deterministic CLI for checking commit messages, raw message files,
  stdin, Git ranges, and full-history audits.
- Make the default policy useful without configuration.
- Let teams configure stricter or looser authorship/provenance rules.
- Return stable structured violations that can power CI and annotations.
- Support deployment in local hooks, CI required checks, GitHub rulesets, and
  merge-bot flows.
- Fail closed when required metadata is missing or malformed.

**Non-Goals:**

- Detecting whether code itself was generated by AI.
- Rewriting historical Git commits.
- Hosting a SaaS policy service.
- Replacing GitHub/GitLab branch protection.
- Making legal determinations about past authorship disputes.

## Threat Model

The MVP is designed for these actors and failure modes:

- Default-behavior tools that automatically append authorship-like markers.
- Default-behavior tools that create commits using tool-controlled Git author or
  committer identities.
- Contributors who accidentally paste tool credit markers into commits, pull
  requests, or merge messages.
- Cloud agents that create commits or pull requests outside a developer's local
  Git hooks.
- Platform merge flows where the final protected-branch commit message differs
  from the messages checked by CI.

The MVP is not designed to fully defeat a determined bypass attempt. Examples
include Unicode homoglyph spoofing, deliberately split markers, administrator
rule bypass, or direct protected-branch writes. These evasions should be
documented so users understand the enforcement boundary.

## Decisions

### Use Rust native CLI for the MVP

The project will use Rust and Cargo for the initial CLI.

Rationale:

- `creditlint` is a low-level workflow tool that should be easy to install in
  projects that do not already have a JavaScript toolchain.
- A single native binary works well for Git hooks, CI jobs, merge bots, and
  repository policy automation.
- Rust gives strong types, predictable error handling, fast startup, and good
  libraries for CLI parsing, regex, YAML, and structured output.
- crates.io, GitHub Releases, Homebrew, and optional npm wrapper packages can
  cover broad distribution without making Node.js a runtime requirement.

Alternative considered: TypeScript/npm first. That would speed up npm
publication, but it would make the default installation path depend on a
toolchain many target repositories do not otherwise need.

### Keep Rust tooling small and standardized

The repository should include a minimal Rust tooling baseline:

- `rust-toolchain.toml` to pin the stable channel and required components
  (`rustfmt`, `clippy`).
- `just` for a small, readable command runner around common Cargo workflows.
- `cargo-nextest` for fast, reliable test execution in CI and local development.
- `cargo-watch` as an optional local development helper.
- `cross` only for release builds that need cross-platform artifacts.

The repository should not require `bacon` or `cargo-edit` for normal
development. They are useful personal tools, but they should not become project
requirements for the MVP.

Rationale:

- `just` makes commands discoverable without hiding Cargo.
- `cargo-nextest` improves test speed and reporting once integration tests create
  temporary Git repositories.
- `cargo-watch` and `bacon` overlap; documenting `cargo-watch` as optional keeps
  the baseline lighter.
- `cargo-edit` helps edit dependencies interactively, but dependency changes
  should still be explicit in `Cargo.toml` review.
- `cross` is valuable for release packaging, but not needed for the first local
  development loop.

Alternative considered: require the full Rust CLI tool stack from the start.
That would make contributor setup heavier before the project has enough code to
benefit from it.

### Keep the policy engine independent from the CLI

The core checker should expose functions that accept a message, policy, and
source metadata, then return structured violations. CLI commands should only
handle input collection, output formatting, and exit codes.

Rationale:

- The same checker must serve local hooks, CI, audits, and merge-bot validation.
- Tests can exercise policy behavior without shelling out to Git.
- Future integrations can reuse the checker without duplicating policy logic.

Alternative considered: implement all logic directly in command handlers. That
would be faster for the first command but harder to test and extend.

### Treat GitHub squash merge as a platform boundary

`creditlint check --range` can validate PR commits, but it cannot see a final
squash commit message edited in GitHub UI. The tool should support that path by
generating GitHub ruleset-compatible patterns and validating final merge-message
files for merge bots.

Rationale:

- CI checks inspect proposed inputs; repository rulesets or merge bots control
  the final protected-branch write.
- This separation avoids overstating what a normal CI job can guarantee.

Alternative considered: claim CI is enough for squash merge. That is not
accurate when the final message can differ from the checked commits.

### Keep GitHub ruleset export conservative

`creditlint github ruleset-pattern` should export only the subset of policy that
can be represented safely as one GitHub commit-message regex. When a configured
policy uses allowlists, rule precedence, Unicode normalization, or other logic
that cannot be faithfully represented, export must fail closed and point users
to merge-bot validation.

Rationale:

- A lossy ruleset regex would create false confidence in final squash-message
  enforcement.
- GitHub rulesets are strongest for simple "message must not match this marker"
  policies.
- Merge bots are the correct path for policies that need full `creditlint`
  evaluation of the final message.

Alternative considered: generate a best-effort regex for every policy. That
would be misleading because unsupported policy semantics could silently be lost.

### Default policy blocks AI/tool authorship markers, not all co-authors

The default policy should reject AI/tool identities in both Git identity metadata
and authorship-like message fields. For message trailers, it should reject
`Co-authored-by` only when its value looks like an AI tool, bot, or coding-agent
vendor. Human authors and co-authors should remain allowed unless the project
config chooses a stricter rule.

Rationale:

- `Co-authored-by` is legitimate for human collaboration.
- Git `author` and `committer` metadata are first-class authorship signals and
  must be checked separately from commit message text.
- The immediate risk is unauthorized tool identity, authorship, or credit
  markers.
- Teams can still configure "no co-authors" or allowlist-only modes.

Alternative considered: block every `Co-authored-by` trailer by default. That is
safer for a narrow policy but too disruptive for common GitHub workflows.

### Use conservative text parsing first, structured trailer parsing later if needed

The MVP should detect trailer-like lines and common marker lines directly in
Rust. It should keep the parser structured enough to report fields, line
numbers, and rule IDs. If Git trailer compatibility becomes a problem, a later
change can shell out to `git interpret-trailers` or port stricter parsing.

Rationale:

- The forbidden markers are simple and should be detected even outside a formal
  trailer block.
- Shelling out for every message can be slower and less portable in tests.
- A structured internal model is enough for MVP reporting.

Alternative considered: rely entirely on `git interpret-trailers`. That is
accurate for trailers but insufficient for `Made with ...` lines outside trailer
blocks.

### Match free-form markers narrowly

Free-form marker detection should be line-oriented and conservative. Default
free-form rules should match explicit marker lines such as `Made with Cursor`,
not prose such as `The fix was made with care`.

Rationale:

- Commit bodies often contain ordinary English phrases that include "made with"
  or "generated with".
- False positives in CI will make maintainers disable the tool.
- Explicit marker-line matching still catches the high-risk default output from
  coding agents.

Alternative considered: match forbidden phrases anywhere in the message. That
is simpler but too noisy for commit bodies and pull request bodies.

## Risks / Trade-offs

- [Risk] Regex policies can be too broad and flag legitimate text. -> Mitigate
  with clear rule IDs, JSON output, and config overrides.
- [Risk] GitHub ruleset regex cannot represent every custom policy. -> Mitigate
  by making `github ruleset-pattern` fail when a policy cannot be safely
  represented.
- [Risk] Existing Git hooks may be overwritten. -> Mitigate by refusing to
  overwrite unmanaged hooks unless a future explicit force flag is added.
- [Risk] CI users may assume `check --range` covers final squash messages. ->
  Mitigate with docs and a separate workflow-integration requirement.
- [Risk] Default AI/vendor keyword lists can become stale. -> Mitigate by
  documenting them as defaults and making them configurable.
- [Risk] Large repository audits may be slow. -> Mitigate with streaming Git
  metadata collection and an explicit performance budget before release.
- [Risk] Users may worry that commit messages are uploaded. -> Mitigate with a
  documented no-telemetry/no-network default.

## Migration Plan

This is a new project, so there is no runtime migration. Implementation should
land in stages:

1. Bootstrap Cargo package metadata, rust-toolchain, just recipes, Rust
   build/test tooling, and CLI entrypoint.
2. Implement the policy engine with unit tests.
3. Add `check` input modes and output formats.
4. Add Git range/audit commands that collect identity metadata and messages,
   plus integration tests using temporary repos.
5. Add `init`, hook installation, and CI examples.
6. Add PR title/body text-file validation documentation.
7. Add GitHub ruleset pattern generation after core behavior is stable.
8. Add release packaging with GitHub Releases and cross-platform builds.

Rollback strategy is simple before first release: revert the change or adjust
the active OpenSpec proposal before implementation is archived.
